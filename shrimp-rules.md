# Development Guidelines

## Project Scope
- 이 저장소는 Vercel에서 매일 오전 9시(KST) 자동 실행되는 뉴스 수집/분석/메일 발송 파이프라인을 구축한다.
- 기존 기준 구현은 `C:\Codex\0219_NewsCollectorV2 (2)`이며, 수집 로직·분석 흐름·메일 포맷은 가능한 한 동일하게 유지한다.
- 대시보드나 수신자 관리 웹 UI는 이번 범위에서 제외하고, 배치 실행과 메일 발송 성공률을 우선한다.

## Architecture Rules
- 수집기, 필터, 분석기, 발송기 구조는 기존 프로젝트의 역할 분리를 유지하되, Vercel에 맞는 서버리스 진입점으로 재구성한다.
- Vercel Cron이 호출하는 엔드포인트 또는 함수는 오케스트레이션만 담당하고, 핵심 도메인 로직은 재사용 가능한 모듈에 둔다.
- 파일 시스템 영속 저장이나 로컬 HTML 대시보드 생성에 의존하지 않는다. 실행 결과는 메일 본문과 로그 중심으로 처리한다.

## Platform Rules
- 스케줄 기준 시각은 한국 시간 오전 9시이며, Vercel 설정에서는 UTC 변환을 명시적으로 관리한다.
- 비밀 정보는 Vercel 환경 변수로만 주입하며 `.env.example`에는 키 이름만 남긴다.
- 장시간 실행, 브라우저 의존, 로컬 상태 저장이 필요한 설계는 피하고 HTTP/API 중심 경량 실행을 우선한다.

## Required Sync Rules
- 환경 변수 추가/변경 시 아래를 반드시 동시에 갱신한다.
  - `.env.example`
  - `README.md`
  - 런타임 설정 모듈
- 스케줄 또는 실행 진입점 변경 시 아래를 반드시 함께 갱신한다.
  - `vercel.json`
  - 엔트리 함수 파일
  - `README.md`
- 메일 발송 계약이나 수신자 구조를 바꾸면 관련 포매터/발송기/문서를 함께 수정한다.

## Migration Rules
- 기존 프로젝트의 웹 전용 코드(`web/`, `start_web.py`, 웹 템플릿)는 기본적으로 이관 대상에서 제외한다.
- 기존 프로젝트의 수집기/필터/분석기/메일 포매터는 우선 복제 또는 최소 수정 이관을 검토하고, 필요 시에만 Vercel 제약에 맞게 변경한다.
- 외교부 0404 알림 등 기존 자동 메일 요구사항이 유지되는지 구현 전 확인하고, 제외 시 README에 명시한다.

## Validation Rules
- 최소 검증 범위는 로컬 단일 실행 성공, 주요 모듈 import 성공, 환경 변수 누락 시 명확한 에러 반환이다.
- 스케줄 실행 경로는 로컬에서 HTTP 엔드포인트 호출 또는 함수 직접 호출로 재현 가능해야 한다.
- 메일 발송은 실서버 의존 없이 dry-run 또는 formatter 단위 테스트를 우선 제공한다.

## AI Decision Rules
- 새로운 구조를 처음부터 다시 설계하기보다, 기존 프로젝트의 검증된 로직을 최대한 재사용하는 방안을 우선한다.
- Vercel 제약과 충돌하는 기능은 제거 이유와 대체 방식을 문서에 남긴다.
- 사용자 사용법, 배포 방식, 환경 변수에 영향이 있으면 반드시 `README.md`를 갱신한다.

## Prohibited Actions
- 기존 동작을 조용히 축소하거나 누락하지 않는다. 제외된 기능은 명시적으로 기록한다.
- 전역 shrimp 설정을 수정해 다른 프로젝트에 영향 주는 방식은 사용하지 않는다.
- 대시보드 제거 요구가 있는 상태에서 불필요한 UI 레이어를 추가하지 않는다.
